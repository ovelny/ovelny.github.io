personal debian setup

average setup time: ~3 hours

this document is a step-by-step process to reproduce the setup i like for a fresh debian install. as such, it is unlikely to fit your needs and preferences and is mainly aimed for me. it also involves a number of dotfiles and settings that are kept private.

sections marked as deprecated at the end of this document don't mean that the tools / apps mentioned are out-of-date. it means i'm not using them anymore, but prefer to keep those notes around just in case.

some other sections might begin with a todo note, meaning they are being updated.

enable debian backports and edit sources.list

debian backports are useful to get the latest packages for non-critical tools. to install it, follow the instructions on https://backports.debian.org/Instructions/ which are simply:

  • add the following lines in /etc/apt/sources.list:
    • deb http://deb.debian.org/debian bookworm-backports main (check that bookworm is indeed your current version)
  • run sudo apt update and sudo apt upgrade

to search / install a package from the backports, just add the flag -t bookworm-backports to your usual apt commands:

  • sudo apt install -t bookworm-backports <package-name>
  • apt search -t bookworm-backports <package-name>

in case your sources.list file gets messed up, refer to the examples here: https://wiki.debian.org/SourcesList#Example_sources.list

make sure that every line except the backports one in sources.list uses main contrib non-free non-free-firmware.

configure DNS

i want the following:

  • without using a VPN, use quad9 nameservers (or any others you want)
  • with a VPN active, let it use its own DNS

to make this work, follow those steps:

# first make sure your VPN is off, then:

# ensure resolvconf is installed:
apt search resolvconf

# append the following lines in /etc/resolvconf/resolv.conf.d/head:
nameserver 9.9.9.9
nameserver 149.112.112.112

# disable and stop systemd-resolved, if present:
sudo systemctl disable systemd-resolved.service
sudo systemctl stop systemd-resolved

# add the following line in the [main] section of /etc/NetworkManager/NetworkManager.conf:
dns=none

# edit /etc/resolv.conf with the same settings added to resolvconf:
nameserver 9.9.9.9
nameserver 149.112.112.112

# remove other nameserver lines if there are any

# reboot:
sudo poweroff --reboot

# ensure everything works by switching your VPN on/off, and check the content of /etc/resolv.conf. you can make sure this is working with ipleak.net too, for instance.

enable ping from guest machines in QEMU

permissions need to be enabled to ping from your attack box:

# get your current user's GID (will likely be 1000)
id

# give ping permissions to this gid by adding
# the following line in /etc/sysctl.conf
net.ipv4.ping_group_range = <gid> <gid> # replace <gid> with number found with 'id'

# apply changes
sudo /sbin/sysctl -p

restore dotfiles

start by restoring dotfiles: we'll have to launch syncthing manually once, before the autostart file takes over after this step.

run syncthing (remove default folder too) and start syncing the ~/.dotfiles directory. in advanced options, choose to ignore syncing permissions.

you can now reboot. the updated zsh theme and automatic syncthing startup should be proof enough that your dotfiles are restored.

if you are stuck in a login loop, always going back to the prompt screen, this might be due to my current config and is nothing to fear: press ctrl + alt + F2 and login in tty, then type exit to quit tmux and use startx.

this workaround is only temporary: finishing the present setup fixes it.

one last thing: because we ignore syncing permissions, we have to restore the execute bit for all scripts in ~/bin. this can be done easily with cd ~/bin && while read f; do chmod +x "$f"; done < <(find . -type f ! -name "*.*")

screenshots

gnome being gnome, they're having a blast changing every 3 years how screenshots are taken. i got extremely tired of this.

there is only one way i take screenshots: selecting an area and pasting it into the clipboard. because accidentally entering a shorcut should never, ever lead to a random screenshot being created somewhere. this, should remain under your control.

thankfully i stumbled on several tools (unix-style, doing only one thing) that provide exactly what i need:

  • sx4, selection tool for X11
  • sxot, screenshot tool for X11
  • optipng with optipng pipe, to convert into an optimized .png

this all lead to the following command for my use case:

sxot -g $(sx4 2>/dev/null) | optipng-pipe | xclip -selection clipboard -target image/png

everything is almost set up for this. sx4, sxot and optipng-pipe are already all in ~/bin, and optipng should be installed on a default debian system. the command above is already in the ~/bin/screenshot wrapper.

all you need to do is to set up a custom shortcut in gnome, binding PrtScn to /home/ovelny/bin/screenshot. that's all.

should you need to recompile sxot and sx4 for a different architecture, you already copied them you-know-where.

should you need to output your clipboard's content to a .png file, you have your ~/bin/clipste script for this.

fspdf installation

fspdf is a fantastic tool to fill and sign PDFs. with less than 500 LOCs, this python app manages to be infinitely more intuitive and reliable than okular or others for this task.

the app only needed to be updated for recent versions of pillow. clone the fork with git clone git@github.com:ovelny/fspdf.git, go to the directory and install it with pipx install ..

system dependencies should already be there, but in case of issues check the repo for information. another thing needed for the app to work is to edit imagemagick's security policy for PDFs: in /etc/ImageMagick-{version}/policy.xml, find the following line:

<policy domain="coder" rights="none" pattern="PDF" />

and replace it with:

<policy domain="coder" rights="read | write" pattern="PDF" />

fspdf should now be set up and working.

automount drives

set your secondary hard drive and others to automount: when using gnome-disks, this is simply done by selecting your drive -> click on settings -> edit mount options -> disable user session defaults. reboot to check this is working once done.

the process is the same with encrypted drives, go to encryption options -> disable user session defaults -> set auto-decryption at startup and reboot to check if it works. if it does, go to the now decrypted partition and set automount as described in previous paragraph.

keep in mind that permissions might be messed up, and you might have to reassign them with take ownership setting (always in gnome-disks), recursively or not.

restic config and restore

restic's "key" is already present in ~/.dotfiles, and its symlink can be found in ~/.config/restic/restic_key.gpg after running the previous steps.

you can use that key to decrypt restic's repo with the following flag:

restic --password-command "gpg --decrypt --default-recipient-self /home/ovelny/.config/restic/restic_key.gpg" <rest-of-the-command>...

ensure that your restic repository is intact by running restic --password-command "<...>" -r /<repo-path> check.

since we're aiming for a fresh install, we're not going to restore the entire /home directory. only the following should be restored:

  • ~/Documents
  • ~/qemu
  • ~/Videos
  • ~/roms

the rest is either already present thanks to syncthing or located in the second drive — at the time of writing this document.

the second drive — whatever its location is — can be restored entirely with the exception of the Steam directory. that one can be kept away unless you need to restore a save someday.

use the following to restore a specific file or directory with restic, which should also work nicely for the entire 2nd drive:

# grab the ID of the snapshot you wanna use
restic --password-command "gpg --decrypt --default-recipient-self /home/ovelny/.config/restic/restic_key.gpg" -r /<path-of-restic-repo> snapshots 

# restore a directory or file
restic --password-command "gpg --decrypt --default-recipient-self /home/ovelny/.config/restic/restic_key.gpg" -r /<path-of-restic-repo> restore <ID> --target /<where-to-restore-directory> --include /<path-of-directory-to-restore>

keep in mind however that the absolute path is restored, not only the directory's content. this is how restic works and there is currently no way around it (except for mounting restic's repo somewhere else, but i don't like that). just mv the files accordingly and remove the emptied absolute path.

sync main directories with syncthing

i use syncthing with the following configuration:

computer 1 <-> smartphone <-> computer 2

computers, no matter their number, never talk directly between each other. everything i sync goes through my smartphone, who acts as a hub.

the following directories should be synced with syncthing:

directory phone trusted / untrusted? label already done?

/home/ovelny/Pictures trusted default no
/home/ovelny/bin untrusted default yes

for all directories, make sure to disable syncing permissions in advanced options and disable file versioning.

for directories that are not already synced on the other end, delete the default directory if it already exists (like ~/Pictures). that way you can be sure you won't sync anything the other way around, from computer 2 to computer 1 — it's a bit brutal but it feels safer. especially with an untrusted device holding encrypted data between two machines.

a few words of advice with android use:

  • pick the syncthing-fork app on f-droid.
  • use the normal interface to watch for changes / sync and restart the service.
  • use the web interface for everything else, especially configuration.
  • if you must sync large files, try to connect your machine and phone on the same local network and allow syncthing in ufw with sudo ufw allow syncthing. a direct QUIC connection between devices is immensely better.
  • even simpler, use your syncthingufw script to automate the ufw rules above.

local SSH config

i use a local SSH config to sync some large files with rsync. nearly everything is taken care of once the following is done:

  • cat ~/.ssh/config and see what addresses are bound to the local machines
  • set up static DHCP addresses on your router, binding the addresses to the machines as shown in the SSH config
  • copy your sshd_config (you know where to find it) in /etc/ssh/sshd_config
  • change ListenAddress line in sshd_config so it matches your machine's address
  • restart the SSH server with sudo systemctl restart ssh

syncing with rsync is already configured with city_ruins_sync in ~/bin. even if the sshd_config is pretty hardened, we have openlocalssh and closelocalssh aliases to open/close the port easily before/after syncing what we need.

it's important that the SSH service remains disabled, as our config listen to local addresses that aren't present yet during the boot process:

sudo systemctl disable ssh && sudo systemctl stop ssh

    // boot the server and start SuperDirt

        ~dirt.doNotReadYet = true; // load samples on demand, memory saving
        ~dirt.loadSoundFiles;   // load samples (path containing a wildcard can be passed in)

        ~dirt.start(57120, 0 ! 12);   // start listening on port 57120, create two busses each sending audio to channel 0

sudo tar xzvf /tmp/opensnitch-settings.tar.gz --directory=/etc/opensnitchd
sudo mv /etc/opensnitchd/etc/opensnitchd/* /etc/opensnitchd
sudo rm -rf /etc/opensnitchd/etc
